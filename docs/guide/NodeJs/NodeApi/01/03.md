# Node.js 事件循环机制

## 1. 事件循环概述

1. `Node.js` 是单线程的，但通过事件循环机制实现了高并发处理能力。
2. 事件循环是一个无限循环，负责收集和执行各种异步任务的回调函数。
3. 事件循环的核心目标是处理非阻塞的 `I/O` 操作，将操作结果分派给相应的回调函数。

```js
// 事件循环伪代码
while (true) {
  // 检查是否有待处理的任务
  if (hasPendingTasks()) {
    // 执行任务回调
    executeCallback()
  }
}
```

## 2. 宏任务与微任务

### 宏任务（Macro Task）

1. 宏任务是较大的任务单元，每个宏任务都有自己的执行上下文。
2. 常见的宏任务包括：

- `setTimeout`
- `setInterval`
- `setImmediate`（Node.js 特有）
- `I/O` 操作
- `UI` 渲染（浏览器环境）
- `script` 整体代码执行

### 微任务（Micro Task）

1. 微任务是较小的任务单元，在当前操作执行完毕后立即执行。
2. 微任务优先级高于宏任务，在事件循环的特定节点执行（详见下方执行顺序）。
3. 常见的微任务包括：

- `Promise.then/catch/finally`
- `process.nextTick`（Node.js 特有）
- `queueMicrotask`
- `MutationObserver`（浏览器环境）

### 执行顺序

1. 同步代码执行完毕后，先清空微任务队列（`process.nextTick` > `Promise`）。
2. 进入事件循环，在各阶段中执行宏任务回调。
3. 微任务执行时机（Node.js v11+ 行为）：
   - `timers` 阶段：每执行完一个定时器回调，立即清空微任务队列。
   - `check` 阶段：每执行完一个 `setImmediate` 回调，立即清空微任务队列。
   - 其他阶段（如 `poll`）：阶段内回调执行完毕后，清空微任务队列。
4. `process.nextTick` 的优先级高于 `Promise`：

- `process.nextTick` 会在当前操作完成后立即执行。
- `Promise` 微任务在 `process.nextTick` 队列清空后执行。

```js
console.log('1. 同步代码')

setTimeout(() => {
  console.log('2. setTimeout 宏任务')
}, 0)

Promise.resolve().then(() => {
  console.log('3. Promise 微任务')
})

process.nextTick(() => {
  console.log('4. process.nextTick 微任务（优先级最高）')
})

console.log('5. 同步代码结束')

// 输出顺序：
// 1. 同步代码
// 5. 同步代码结束
// 4. process.nextTick 微任务（优先级最高）
// 3. Promise 微任务
// 2. setTimeout 宏任务
```

## 3. 事件循环的六个阶段

事件循环从一次循环到下一次循环，会经历以下六个阶段：

### 阶段一：timers（定时器阶段）

1. 执行 `setTimeout` 和 `setInterval` 的回调。
2. 检查是否有到期的定时器，如果有则执行其回调。
3. 注意：`setTimeout` 的延迟时间只是最小等待时间，实际执行时间可能更长。
4. 从 Node.js v11 开始，每执行完一个定时器回调，都会立即清空微任务队列。

```js
setTimeout(() => {
  console.log('timers 阶段执行')
}, 100)
```

### 阶段二：pending callbacks（待处理回调阶段）

1. 执行上一轮循环中未执行的 `I/O` 回调。
2. 处理一些系统操作的回调，如 `TCP` 错误等。
3. 大多数回调不会在这个阶段执行，除非特殊情况。

```js
const net = require('net')
const server = net.createServer(() => {}).listen(8080, () => {
  // 服务器启动回调可能在 pending callbacks 阶段执行
})
```

### 阶段三：idle, prepare（空闲、准备阶段）

1. 仅内部使用，开发者无法干预。
2. 用于系统内部的准备工作。

### 阶段四：poll（轮询阶段）

1. 这是事件循环中最重要的阶段。
2. 主要功能：

- 执行 `I/O` 回调（如文件读取、网络请求等）。
- 处理 `poll` 队列中的事件。

3. 执行逻辑：

- 如果 `poll` 队列不为空，则同步执行所有回调，直到队列为空。
- 如果 `poll` 队列为空：
  - 如果有 `setImmediate` 回调，则进入 `check` 阶段。
  - 如果没有 `setImmediate` 回调，则等待新的 `I/O` 事件加入队列。

```js
const fs = require('fs')

fs.readFile('./test.txt', 'utf8', (err, data) => {
  console.log('poll 阶段执行 I/O 回调')
})
```

### 阶段五：check（检查阶段）

1. 执行 `setImmediate` 的回调。
2. `setImmediate` 设计为在 `poll` 阶段完成后立即执行。
3. 从 Node.js v11 开始，每执行完一个 `setImmediate` 回调，都会立即清空微任务队列。
4. 与 `setTimeout(fn, 0)` 的区别：

- `setImmediate` 在 `poll` 阶段后执行。
- `setTimeout(fn, 0)` 在 `timers` 阶段执行。

```js
setImmediate(() => {
  console.log('check 阶段执行')
})
```

### 阶段六：close callbacks（关闭回调阶段）

1. 执行关闭事件的回调，如 `socket.on('close', ...)`。
2. 处理一些资源清理工作。

```js
const server = require('net').createServer()
server.on('close', () => {
  console.log('close callbacks 阶段执行')
})
```

## 4. 事件循环完整流程图

```
   ┌───────────────────────────┐
┌─>│           timers          │  执行 setTimeout/setInterval 回调，每个回调后清空微任务
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │     pending callbacks     │  执行上一轮未执行的 I/O 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │       idle, prepare       │  内部使用
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │           poll            │  执行 I/O 回调，处理 poll 队列，阶段结束后清空微任务
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │           check           │  执行 setImmediate 回调，每个回调后清空微任务

└──│      close callbacks      │  执行关闭回调
   └───────────────────────────┘

   注：Node.js v11+ 中，timers 和 check 阶段每执行一个回调后就清空微任务队列
   
```

## 5. 微任务执行时机

1. 微任务在以下时机执行（Node.js v11+ 行为）：

- `timers` 阶段：每执行完一个定时器回调后立即清空微任务队列。
- `check` 阶段：每执行完一个 `setImmediate` 回调后立即清空微任务队列。
- `poll` 阶段：阶段内所有 I/O 回调执行完毕后，清空微任务队列。
- 其他阶段：阶段结束时清空微任务队列。
- 事件循环启动前（同步代码执行完后），也会清空微任务队列。

2. `process.nextTick` 和 `Promise` 的区别：

- `process.nextTick` 不属于事件循环的任何阶段。
- `process.nextTick` 会在当前操作完成后立即执行，优先级高于其他微任务。
- 过度使用 `process.nextTick` 可能导致 `I/O` 饥饿。

```js
// process.nextTick 优先级高于 Promise
process.nextTick(() => {
  console.log('nextTick 1')
  process.nextTick(() => {
    console.log('nextTick 2')
  })
})

Promise.resolve().then(() => {
  console.log('Promise 1')
})

// 输出：
// nextTick 1
// nextTick 2
// Promise 1
```

## 6. 具体案例分析

### 案例一：setTimeout 与 setImmediate

```js
setTimeout(() => {
  console.log('setTimeout')
}, 0)

setImmediate(() => {
  console.log('setImmediate')
})
```

**分析**：

- 执行顺序不确定，取决于事件循环启动时是否已进入 `timers` 阶段。
- 如果启动时直接进入 `timers` 阶段，`setTimeout` 先执行。
- 如果启动时先进入其他阶段，`setImmediate` 可能先执行。

### 案例二：在 I/O 回调中调用

```js
const fs = require('fs')

fs.readFile(__filename, () => {
  setTimeout(() => {
    console.log('setTimeout')
  }, 0)
  setImmediate(() => {
    console.log('setImmediate')
  })
})
```

**分析**：

- 在 `I/O` 回调中，执行顺序是确定的。
- `I/O` 回调在 `poll` 阶段执行。
- `poll` 阶段结束后进入 `check` 阶段，执行 `setImmediate`。
- 下一轮循环的 `timers` 阶段才执行 `setTimeout`。
- **输出顺序**：`setImmediate` → `setTimeout`

### 案例三：综合案例

```js
console.log('start')

setTimeout(() => {
  console.log('setTimeout 1')
  Promise.resolve().then(() => {
    console.log('promise 1')
  })
}, 0)

setTimeout(() => {
  console.log('setTimeout 2')
}, 0)

Promise.resolve().then(() => {
  console.log('promise 2')
})

process.nextTick(() => {
  console.log('nextTick 1')
  process.nextTick(() => {
    console.log('nextTick 2')
  })
})

setImmediate(() => {
  console.log('setImmediate')
})

console.log('end')
```

**执行流程分析**（Node.js v24.x）：

```
1. 同步代码执行：
   - 打印 'start'
   - 打印 'end'

2. 事件循环启动前，清空微任务队列（按优先级）：
   - 打印 'nextTick 1'
   - 打印 'nextTick 2'
   - 打印 'promise 2'

3. timers 阶段（第一轮）：
   - 执行第一个 setTimeout 回调：打印 'setTimeout 1'
     - 该回调添加了 Promise 微任务，v24.x 下立即清空微任务：打印 'promise 1'
   - 执行第二个 setTimeout 回调：打印 'setTimeout 2'

4. check 阶段（第一轮）：
   - 打印 'setImmediate'
```

**最终输出**：

```
start
end
nextTick 1
nextTick 2
promise 2
setTimeout 1
promise 1
setTimeout 2
setImmediate
```

> **注意**：在主模块中，`setTimeout(fn, 0)` 与 `setImmediate` 的执行顺序本身是不确定的，取决于事件循环启动时 `timers` 阶段是否已到达。上述分析描述的是通常情况下（`timers` 阶段先于 `check` 阶段执行）的执行顺序。

### 案例四：嵌套微任务

```js
process.nextTick(function A() {
  console.log('A')
  process.nextTick(function B() {
    console.log('B')
  })
})

Promise.resolve().then(function C() {
  console.log('C')
  process.nextTick(function D() {
    console.log('D')
  })
})
```

**分析**：

- 微任务队列会优先清空所有 `process.nextTick`。
- `A` 执行时又添加了 `B`，`B` 会在 `C` 之前执行。
- `C` 执行时添加了 `D`，`D` 会在下一轮微任务中执行。

**输出**：

```
A
B
C
D
```

### 案例五：I/O 与定时器结合

```js
const fs = require('fs')

console.log('1. 开始')

fs.readFile('./test.txt', 'utf8', (err, data) => {
  console.log('2. 文件读取完成')
  process.nextTick(() => {
    console.log('3. nextTick in I/O')
  })
  Promise.resolve().then(() => {
    console.log('4. Promise in I/O')
  })
})

setTimeout(() => {
  console.log('5. setTimeout')
}, 0)

setImmediate(() => {
  console.log('6. setImmediate')
})

console.log('7. 结束')
```

**执行流程**（Node.js v24.x，假设文件 I/O 在第一轮 poll 阶段完成）：

```
1. 同步代码：打印 '1. 开始' 和 '7. 结束'
2. 事件循环启动前，清空微任务队列（此处无微任务）
3. timers 阶段：打印 '5. setTimeout'
4. poll 阶段：
   - 执行 I/O 回调：打印 '2. 文件读取完成'
   - 回调执行完后立即清空微任务队列：
     - 打印 '3. nextTick in I/O'
     - 打印 '4. Promise in I/O'
5. check 阶段：打印 '6. setImmediate'
```

**可能的输出**：

```
1. 开始
7. 结束
5. setTimeout
2. 文件读取完成
3. nextTick in I/O
4. Promise in I/O
6. setImmediate
```

> **注意**：
> 1. 在主模块中，`setTimeout(fn, 0)` 与 `setImmediate` 的执行顺序是不确定的。如果事件循环启动时尚未进入 `timers` 阶段，`setImmediate` 可能先于 `setTimeout` 执行。
> 2. 文件 I/O 的完成时机也不确定，如果 I/O 在第一轮 `poll` 阶段尚未完成，则 `setImmediate` 可能在 I/O 回调之前执行。上述分析仅为最常见的情况。

## 7. 与浏览器事件循环的区别

| 特性 | Node.js | 浏览器 |
|------|---------|--------|
| 微任务执行时机 | timers/check 阶段每个回调后执行；poll 阶段结束时执行 | 每个宏任务之后 |
| 特有 API | `process.nextTick`、`setImmediate` | `requestAnimationFrame`、`MessageChannel` |
| 阶段数量 | 6 个阶段 | 2 个阶段（宏任务、微任务） |
| `setTimeout` 执行 | `timers` 阶段，每个回调后清空微任务 | 宏任务队列，宏任务后清空微任务 |
| `setImmediate` 执行 | `check` 阶段，每个回调后清空微任务 | 无对应 API |

## 8. 最佳实践

1. **避免过度使用 `process.nextTick`**：可能导致 `I/O` 饥饿，阻塞其他任务。
2. **合理使用 `setImmediate`**：适合在 `I/O` 操作后立即执行的场景。
3. **理解执行顺序**：对于调试异步代码非常重要。
4. **注意错误处理**：异步回调中的错误需要正确捕获，避免导致进程崩溃。

```js
// 错误示例：可能导致 I/O 饥饿
function ioStarvation() {
  process.nextTick(ioStarvation)
}
ioStarvation() // 其他任务永远无法执行

// 正确示例：使用 setImmediate 让出控制权
function healthyAsync() {
  setImmediate(healthyAsync)
}
healthyAsync() // 允许其他任务执行
```
