# Node 模块化与模块化开发

## 1. 给 Node 传递参数

1. 正常情况运行`node`程序，只需要在`node`命令后加文件名就行。

```js
node fileName.js
```

2. 给`node`传递参数，只需要在文件名后面写入参数，再执行即可，该命令传入的参数为`yangzebo`和`age=18`。

```js
node fileName.js yangzebo age=18
```

3. 从程序中获得传入的参数，`node`有一个全局对象`process`，这个对象里包含很多信息，其中就有传入的参数。`process.argv`是一个数组，其中`argv[0]` 是`node`的路径，`argv[1]`是程序的路径，剩下的就是你传入的参数。

```js
const argv = process.argv
console.log(argv)
/* ['C:\\Program Files\\nodejs\\node.exe', 
   'E:\\myFile\\ajax\\TestCode\\index.js', 
   'yangzebo', 'age=18'] */
```

4. 为何叫`argv`，`c/c++`程序中的`main`函数中实际上可以获取到两个参数，`argc`与`argv`:

- `argc`，`argument counter`的缩写，表示传递参数的个数。
- `argv`，`argument vector`的缩写，表示传入的具体参数。
- `vector`在程序中表示一种数据结构，在`c++/Java/js`中，都是一种数组结构，里面存储一些参数信息。

## 2. Node 程序的输出

1. `console.log()`，输出传入的参数。

```js
console.log('打印的内容') // 输出 '打印的内容'
```

2. `console.clear()`，清除之前输入的内容。

```js
console.log('前面打印的内容')
console.clear()
console.log('后面打印的内容') // 只会显示 '后面打印的内容'
```

## 3. Node 常用全局对象

### 概述

1. `Node.js` 的全局对象是 `global`，浏览器环境中 `js` 的全局对象是 `window`。后来添加了 `globalThis` 可以指代两个环境中的全局对象。
2. 后面的`process`、`console`、`setTimeout`等都有被放到`global`对象中。
3. 在浏览器中执行的`JavaScript`代码，如果我们在顶级范围内通过`var`定义的一个属性，默认会被添加到`window`
   对象上。但是在`node`中，我们通过`var`定义一个变量，它只是在当前模块中有一个变量，不会放到全局中。

### 特殊与常用

1. 特殊的全局对象，可以直接在所有模块中使用，但是却不能在命令行交互中使用。

- `__dirname`对象，获得当前文件所在文件夹的绝对路径。（不包括当前文件名）
- `__filename`对象，获取当前文件所在的路径。（包括当前文件名）
- `module`对象，`exports`对象，`require`函数

2. 常用的全局对象

- `process`对象，提供了`Node`进程中相关的信息。
- `console`对象，提供了简单的调试控制台。
- `setTimeout`，`setInterval`，`clearTimeout`，`clearInterval`。

## 4. NodeJs 中的模块化

1. `NodeJs`遵循`CommonJs`规范，每个`js`文件都是一个模块，不能直接引用不同模块的数据，需要通过模块中内置的 `exports`，`module.exports`对象导出数据，和`require`引入对象。
2. `exports`是模块中的一个空对象，也是要导出的一个对象。我们在模块中通过`exports.xxx`挂载属性，而在另一个文件中，我们使用`require`引入的就是之前导出的`exports`对象，所以你可以拿到我们之前挂载在对象上的属性。

```js
//  foo.js
var myName = 'yzb',
  age = 18
exports.myName = myName
exports.age = age

//  baz.js
var foo = require('./foo.js')
// 也可以解构赋值
const { myName, age } = require('./foo.js')
foo // { myName: 'yzb', age: 18 }
```

3. 实际上最后的导出操作是`module.exports`导出的，因为在`NodeJS`中源码自动在模块开始给我们做了`module.exports = exports`的操作，导致了他们都指向了同一个内存空间，最后导出的其实是`module.exports`这个对象

```js
// foo.js
// Node.js 在模块加载时做了 module.exports = exports 这个操作
exports.name = 'yzb'
module.exports.name = 'zql'

// baz.js
var foo = require('./foo.js')
foo.name = 'zql' // module.exports 和 exports 指向的是同一片内存空间
```

```js
// foo.js
// Node.js 在模块加载时做了 module.exports = exports 这个操作
exports.name = 'yzb' // {name: 'yzb'}
module.exports = { age: 18 } // 重新赋值一个对象

// baz.js
var foo = require('./foo.js')
foo // { age: 18 } 是 module.exports 这个对象，所以最终导出的是 module.exports
```

## 5. 模块的加载 require

1. 使用`require(xxx)`函数可以在在文件中引入一个文件（模块）中导入的对象。`xxx`可以是

- 内置核心模块
- 文件路径
- 下载的包模块（既不是核心模块也不是路径类似于）

2. `require(核心模块)`，会直接返回核心模块，并且停止查找例如`require(path)`
3. `require(路径)`，例如`require(./xxx)`，`require(../xxx)`。

- 如果路径是一个文件有后缀名，按照后缀名的格式查找对应的文件。如果没有后缀名，会按照如下顺序：
  - 如果只有一个名为`X`的文件，那么对查找该文件，如果文件名相同会按下面顺序查找：
  - 查找`X.js`文件
  - 查找`X.json`文件
  - 查找`X.node`文件
- 如果路径是一个名为`x`的文件夹，则按下面顺序查找目录下的名为`index`的文件
  - 查找`X/index.js`文件
  - 查找`X/index.json`文件
  - 查找`X/index.node`文件
- 找不到则报错

4. `require(第三方包)`，引入一个不是路径也不是核心模块的包。则会先在当前文件夹中的`node_modules`中查找，如果找不到就向外层的`node_modules`中查找，如此循环直到找到导入的模块，如果找不到则会报错。

## 6. 模块加载的过程

1. 模块在被第一次引入时，模块中的`js`代码会被运行一次

```js
// foo.js
console.log('foo')

// main.js
require('./foo')
console.log('main')

// 运行 main.js 后会先打印 foo 再 打印 main，是因为 require 是同步加载的
```

2. 模块被多次引入时，会缓存，最终只加载（运行）一次，因为每个模块的 `module` 对象中都有一个属性 `loaded`，初始值为`false` 当模块执行完毕便会改成`true`，如果第二次或多次引入，会根据`loaded`属性判断此模块已被加载就不会再次加载。

3. 如果多个文件相互引用，那么采用的是数据结构--图结构中的深度优先搜索。

## 7. 模块化规范

### CommonJs

1. `CommonJs`加载模块是同步的，只有等到对应的模块加载完毕，当前模块中的内容才能被运行。
2. 浏览器加载`js`文件需要先从服务器将文件下载下来，之后在加载运行。所以在浏览器中，我们通常不使用`CommonJS`规范。在`webpack`中使用`CommonJS`是另外一回事，因为它会将我们的代码转成浏览器可以直接执行的代码。
3. 在早期为了可以在浏览器中使用模块化，通常会采用`AMD`或`CMD`。

### AMD 规范

1. `AMD` 是 `Asynchronous Module Definition`（异步模块定义）的缩写。
2. 事实上 `AMD` 的规范还要早于 `CommonJS`，但是 `CommonJS` 目前依然在被使用，而 `AMD` 使用的较少了。
3. `AMD` 实现的比较常用的库是 `require.js` 和 `curl.js`。
4. `require.js`使用，[下载地址](https://github.com/requirejs/requirejs)。

- 在`html`页面中使用`script`标签引入`require.js`和定义入口文件。

```html
<script src="./xxx/require.js" data-main="./xxx/index.js"></script>
```

- `data-main`引入的文件就是入口文件，可以引入各个模块。书写规范如下

```js
;(function () {
  require.config({
    baseUrl: '',
    paths: {
      foo: './xxx/foo.js',
      bar: './xxx/bar.js'
    }
  })
  // 引入文件
  require(['foo'], function () {})
  require(['bar'], function () {})
})
```

- 模块的导出

```js
// foo
define(function () {
  let name = 'yzb'
  return {
    name
  }
})
```

- 模块的导入

```js
// bar
define(['foo'], function (foo) {
  console.log(foo.name)
})
```

### CMD 规范

1. `CMD` 是 `Common Module Definition`（通用模块定义）的缩写。
2. 国内 CMD 规范的代表是 Sea.js。
3. 与 AMD 类似，CMD 也是用于浏览器端的异步模块加载方案，现已很少使用。

## 8. ES Modules（ESM）—— Node.js 现代模块系统

### 概述

1. `ES Modules`（简称 ESM）是 `ECMAScript` 官方标准的模块化方案，从 `ES2015`（ES6）开始引入。
2. Node.js 从 **v12** 开始稳定支持 ESM，到 **v24.x** 已经完全成熟，是新项目的推荐选择。
3. ESM 使用 `import` 和 `export` 关键字，语法简洁、静态可分析，支持 Tree Shaking。

### 启用 ESM 的两种方式

**方式一：`package.json` 中设置 `"type": "module"`**

```json
{
  "name": "my-project",
  "type": "module"
}
```

设置后，该目录下所有 `.js` 文件都会被当作 ESM 模块处理。如果想在 `type: "module"` 的项目中使用 CJS，需要将文件后缀改为 `.cjs`。

**方式二：使用 `.mjs` 文件后缀**

`.mjs` 后缀的文件始终被 Node.js 当作 ESM 处理，不受 `package.json` 的 `type` 字段影响。

```bash
node index.mjs
```

同理，`.cjs` 后缀的文件始终被当作 CJS（CommonJS）处理。

### ESM 导出语法

**命名导出（Named Export）**

```js
// math.mjs
export const PI = 3.14159

export function add(a, b) {
  return a + b
}

export class Calculator {
  multiply(a, b) {
    return a * b
  }
}

// 也可以统一导出
const name = 'yzb'
const age = 18
export { name, age }
```

**默认导出（Default Export）**

每个模块只能有一个默认导出：

```js
// user.mjs
export default class User {
  constructor(name) {
    this.name = name
  }
}

// 或者
export default function greet() {
  console.log('hello')
}
```

### ESM 导入语法

**命名导入**

```js
// app.mjs
import { PI, add, Calculator } from './math.mjs'

console.log(PI)          // 3.14159
console.log(add(1, 2))   // 3
```

**默认导入**

```js
import User from './user.mjs'

const u = new User('yzb')
```

**混合导入**

```js
import greet, { PI, add } from './utils.mjs'
```

**命名空间导入（整体导入）**

```js
import * as math from './math.mjs'

console.log(math.PI)
console.log(math.add(1, 2))
```

**重命名导入**

```js
import { add as sum } from './math.mjs'

console.log(sum(1, 2))
```

**只执行模块（不导入任何内容）**

```js
import './init.mjs'  // 仅执行模块中的代码
```

## 9. CJS 与 ESM 的核心区别

| 特性 | CJS（CommonJS） | ESM（ES Modules） |
|------|----------------|-------------------|
| 语法 | `require()` / `module.exports` | `import` / `export` |
| 加载方式 | 同步加载，运行时确定 | 异步加载，编译时静态分析 |
| 模块对象 | `module.exports` 浅拷贝（导出值的拷贝） | `export` 导出的是值的实时绑定（Live Binding） |
| `this` 指向 | 指向当前模块的 `module.exports` | 顶层 `this` 为 `undefined` |
| 顶层 `await` | ❌ 不支持 | ✅ 支持（v14.8+） |
| Tree Shaking | ❌ 难以实现 | ✅ 易于实现 |
| 动态导入 | `require()` 本身就是动态的 | `import()` 函数（返回 Promise） |
| 循环依赖 | 可能拿到不完整的对象 | 支持，通过实时绑定解决 |
| 文件后缀 | `.js` / `.cjs` | `.mjs` 或 `type: "module"` 下的 `.js` |

### 值的实时绑定 vs 值的拷贝

这是 CJS 和 ESM 最重要的区别之一：

```js
// CJS —— 导出的是值的拷贝
// counter.cjs
let count = 0
function increment() {
  count++
}
module.exports = { count, increment }

// main.cjs
const { count, increment } = require('./counter.cjs')
console.log(count)  // 0
increment()
console.log(count)  // 仍然是 0！因为是原始值的拷贝
```

```js
// ESM —— 导出的是值的实时绑定
// counter.mjs
export let count = 0
export function increment() {
  count++
}

// main.mjs
import { count, increment } from './counter.mjs'
console.log(count)  // 0
increment()
console.log(count)  // 1！因为是实时绑定
```

## 10. 动态 import()

1. `import()` 是 ESM 提供的动态导入机制，返回一个 `Promise`。
2. 它可以在代码的任何位置调用，不限于模块顶层。
3. 适用于按需加载、条件加载等场景。

```js
// 按条件加载模块
async function loadLocale(lang) {
  if (lang === 'zh') {
    const zh = await import('./locales/zh.mjs')
    return zh.default
  } else {
    const en = await import('./locales/en.mjs')
    return en.default
  }
}

loadLocale('zh').then(module => {
  console.log(module)
})
```

4. 在 CJS 中使用 `import()` 来加载 ESM 模块（这是 v24.x 中跨系统互操作的关键方式）。

```js
// main.cjs —— 在 CJS 中动态导入 ESM 模块
async function loadConfig() {
  const config = await import('./config.mjs')
  console.log(config.default)
}
loadConfig()
```

## 11. import.meta

1. `import.meta` 是 ESM 中提供模块元信息的对象，仅在 ESM 中可用。
2. `import.meta.url` 返回当前模块的 `file://` URL。

```js
// info.mjs
console.log(import.meta.url)
// 输出类似：file:///Users/yzb/project/info.mjs

import { fileURLToPath } from 'node:url'
import { dirname } from 'node:path'

// 将 file:// URL 转换为文件路径
const __filename = fileURLToPath(import.meta.url)
const __dirname = dirname(__filename)

console.log(__filename)  // /Users/yzb/project/info.mjs
console.log(__dirname)   // /Users/yzb/project
```

3. `import.meta.resolve()` 可以解析模块路径（类似于 CJS 中的 `require.resolve`）。

```js
const moduleUrl = import.meta.resolve('./helper.mjs')
console.log(moduleUrl)  // file:///Users/yzb/project/helper.mjs
```

## 12. CJS 与 ESM 的互操作

### 在 ESM 中导入 CJS 模块

```js
// 在 .mjs 文件中可以直接 import CJS 模块
import cjsModule from './old-module.cjs'
// cjsModule 就是 CJS 模块的 module.exports

import { named } from './old-module.cjs'
// 具名导入也支持，Node.js 会尝试解构 module.exports
```

### 在 CJS 中导入 ESM 模块

```js
// CJS 不能直接 require() ESM 模块
// 但可以使用动态 import() 函数
async function loadESM() {
  const esmModule = await import('./new-module.mjs')
  console.log(esmModule.default)
}
loadESM()
```

## 13. 模块加载过程详解（优化版）

### require 的包装器函数

1. Node.js 在加载 CJS 模块时，并不是直接执行文件内容，而是将模块代码包装在一个函数中：

```js
// Node.js 内部会将你的代码包装成如下形式：
(function(exports, require, module, __filename, __dirname) {
  // 你的模块代码在这里
  const name = 'yzb'
  exports.name = name
})
```

2. 这就是为什么每个模块中可以直接使用 `exports`、`require`、`module`、`__filename`、`__dirname`——它们是包装器函数的参数，而不是真正的全局变量。

### module 对象详解

```js
// 在模块中打印 module 对象
console.log(module)

// Module {
//   id: '.',                    // 模块唯一标识，入口文件为 '.'
//   path: '/Users/yzb/project', // 模块所在目录
//   exports: {},                // 模块导出的对象
//   filename: '/Users/yzb/project/index.js',  // 文件完整路径
//   loaded: false,              // 是否已加载完成
//   children: [],               // 该模块引入的子模块
//   paths: [...]                // 模块搜索路径
// }
```

### 循环依赖的处理

```js
// a.js
console.log('a 开始')
exports.done = false
const b = require('./b.js')
console.log('a 中 b.done =', b.done)
exports.done = true
console.log('a 结束')

// b.js
console.log('b 开始')
exports.done = false
const a = require('./a.js')
console.log('b 中 a.done =', a.done)  // false，因为 a 还未执行完
exports.done = true
console.log('b 结束')

// main.js
const a = require('./a.js')
console.log('main 中 a.done =', a.done)

// 输出顺序：
// a 开始
// b 开始
// b 中 a.done = false   ← a 还未执行完，拿到的是不完整对象
// b 结束
// a 中 b.done = true
// a 结束
// main 中 a.done = true
```

CJS 循环依赖时可能拿到不完整的导出对象。ESM 由于采用实时绑定，可以更好地处理循环依赖。

## 14. Node.js v24.x 模块化最佳实践

1. **新项目推荐使用 ESM**：在 `package.json` 中设置 `"type": "module"`，使用 `import`/`export` 语法。
2. **CJS 转 ESM**：逐步迁移，可通过 `.cjs` 后缀保留旧模块，新模块用 `.mjs` 或 `.js`（在 `type: "module"` 下）。
3. **跨系统互操作**：CJS 中用 `import()` 加载 ESM，ESM 中直接 `import` CJS 模块。
4. **用 `node:` 前缀引用核心模块**（v14.18+ 支持），更清晰地区分核心模块和第三方模块：

```js
// 推荐写法
import fs from 'node:fs'
import path from 'node:path'
```

```js
// CJS 中也推荐
const fs = require('node:fs')
const path = require('node:path')
```

5. **顶层 `await` 仅在 ESM 中可用**，CJS 中无法使用。
